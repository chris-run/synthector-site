What it does

Synthector sits between your content sources and downstream AI/analytics. It transforms text into a safer-by-construction representation and attaches machine-verifiable evidence of how it was produced.

Deterministic redaction

Deterministic detection and replacement of sensitive spans using rule-based signals (regex/dictionaries) plus guarded entity detection.

  • Stable outputs for the same input + ruleset
  • Structured placeholders (e.g., [PERSON_1], [EMAIL])
  • Change control via policy + ruleset identifiers

Posthoc leak-gate

A second pass validates that the anonymized output doesn’t still contain disallowed patterns. Fail-closed behavior is configurable per pipeline.

  • Posthoc leak check runs on anonymized output only.
  • Block, quarantine, or return diagnostics (metadata)
  • Predictable “stop the line” semantics

Attestation + provenance

Output includes metadata-only provenance and optional signed attestation (JWS) so downstream systems can verify policy/ruleset identity and gating outcomes.

  • Ruleset hash + policy ID
  • Gating result + counters
  • Metadata-only provenance (no content)

Tenant isolation + safe aggregates

Isolation boundaries for policies, rulesets, and telemetry. Aggregate-only stats modes for tenant-safe observability.

billing_only health_kmin off
  • Numeric/categorical aggregates only (mode dependent)
  • Policy + ruleset metadata retained

Optional synthetic profiles

Where useful, generate tenant-scoped synthetic replacements without persisting the original or synthetic content. Output only.

  • Non-retained synthetic outputs
  • Useful for demos, QA, and shared analytics

How it works

A deterministic, inspectable pipeline with an explicit zero-retention boundary. Downstream systems receive redacted text plus evidence—without Synthector storing artifacts.

  1. Input
    Text enters from transcripts, meeting notes, chat, or document pipelines.
  2. Detect
    Deterministic detectors identify spans (regex/dict + guarded NER).
  3. Redact
    Sensitive spans are replaced with normalized placeholders or synthetic equivalents (optional).
  4. Leak-check
    Leak-gate validates the anonymized output and can fail closed (block) when required.
  5. Attestation
    Emit metadata-only provenance + optional signed attestation token (JWS).
  6. Output
    Return redacted text, gating result, and verification metadata to your pipeline.
Zero-retention boundary

Processing is performed in-memory. Only policy/ruleset metadata and aggregate counters may be retained (mode-dependent). Synthector processes content in-memory and does not retain customer content artifacts.

Why deterministic matters

Privacy boundaries fail when they’re probabilistic. Determinism gives you reproducible outputs, inspectable change control, and predictable failure modes.

  • Reproducibility: the same input yields the same redacted output given the same ruleset. This enables replay, incident forensics, and regression testing.
  • Auditability: policy + ruleset identifiers (and hashes) make change control explicit. Evidence is attached to output via provenance metadata and optional JWS attestation.
  • Predictable failure modes: leak-gate semantics can fail closed, preventing silent drift or “partial” redaction from flowing downstream.
  • Less LLM surprise behavior: privacy enforcement is not a generative rewrite step. You can still use LLMs downstream—after the boundary.

Security & retention posture

Security posture starts with data minimization. Synthector is designed to avoid becoming a content storage system while still supporting audits and operational health.

Explicit retention statement

Synthector processes content in-memory and does not retain customer content artifacts. Raw and redacted text/audio are not persisted by design.

Artifact Retention Notes
Raw text/audio never stored Processed in-memory; not written to disk by Synthector.
Redacted text/audio never stored Returned to caller; not persisted as an artifact.
Provenance metadata only, TTL-limited Policy ID, ruleset hash, timestamps, gate outcome; no content.
Logs scrubbed, retention-limited Operational logs designed to avoid content; retention is environment/policy controlled.
Aggregates numeric/categorical only Tenant-provisioned mode: billing_only, health_kmin, off. No raw spans; no tokenized content.

Operational semantics

  • Fail-closed where required: leak-gate can block output rather than “best effort”.
  • Tenant isolation: policies/rulesets/telemetry are scoped per tenant boundary.
  • Metadata-only evidence: attestations and provenance are designed for audits without content retention.

Note: This page does not claim SOC 2 / ISO certifications. Synthector is positioned to support your compliance program with deterministic controls and verifiable evidence.

What Synthector is not

Technical overview

This section is intended for ICT / security / platform engineering teams evaluating Synthector as a deterministic privacy boundary in regulated environments.

Architecture

Synthector sits in the ingestion path as a transformation layer. It accepts content, applies a deterministic policy/ruleset, and returns transformed output plus verification metadata. Content processing is performed in-memory.

  • Transform boundary: detect → redact → leak-check
  • Evidence channel: provenance + optional JWS attestation
  • Telemetry channel: aggregate-only counters (mode dependent)

Policy & rulesets

Policies define what must be redacted and what the gate must enforce. Rulesets implement the policy using deterministic detectors (e.g., structured patterns, dictionaries, allowlists/denylists) and guarded entity detection for bounded cases.

  • Explicit policy_id + ruleset versioning
  • Ruleset identity carried as hash for audit/replay
  • Tenant-scoped policy separation

Leak-gate semantics

The leak-gate is a verification step that evaluates only the anonymized output. This reduces leakage pathways and gives you explicit “stop the line” behavior in sensitive environments.

  • Posthoc leak check runs on anonymized output only.
  • Configurable actions: pass, block, or quarantine (metadata-only)
  • Fail-closed option for strict pipelines

Attestation & provenance

Provenance is metadata-only and can be accompanied by a signed attestation token (JWS). Downstream systems can verify that content passed through a specific policy/ruleset and observe gating outcomes.

Retention & telemetry

The retention posture is designed to minimize risk surface: no raw or redacted content artifacts are stored. Telemetry is limited to numeric/categorical aggregates when enabled.

  • Zero retention by default.
  • Aggregate modes: billing_only, health_kmin, off
  • Provenance is metadata-only and TTL-limited

Failure modes & controls

Deterministic transforms allow predictable failure handling. When the system can’t meet policy requirements, it can fail closed and return metadata describing the reason.

  • Block output when residual patterns are detected (strict mode)
  • Regression tests to detect drift in deployed behavior
  • Outcome-level proofs without disclosing detection internals

FAQ

Short answers, technical defaults.

Is this “anonymization” or “pseudonymization”?
It’s a deterministic transformation layer. Depending on policy, outputs may be redaction-style placeholders or tenant-scoped synthetic replacements. Treat outputs as potentially sensitive unless your policy explicitly enforces stronger guarantees for your use case.
Do you store transcripts, raw text, or redacted text?
No. Synthector processes content in-memory and does not retain customer content artifacts. Raw and redacted text/audio are not persisted. Optional telemetry is aggregate-only (numeric/categorical) and mode-controlled.
What does “leak-gate” mean in practice?
It’s a verification pass that checks the anonymized output for residual disallowed patterns and can fail closed (block). Posthoc leak check runs on anonymized output only.
How do partners embed Synthector?
Insert Synthector upstream of ingestion points (LLMs, vector DBs, warehouses). Return redacted output plus provenance/attestation. The goal is to make privacy enforcement an infrastructure concern rather than a per-feature prompt or heuristic.
